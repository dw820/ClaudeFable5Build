AutoCut · End-to-End Architecture

Request flow & where the dynamic workflow lives

One generation, top to bottom. The browser doesn't talk to the agent — it enqueues a run row in Supabase. A long-running worker on Daytona (agent-server.ts) is subscribed via Realtime, claims the run, and runs an autonomous Claude Agent SDK session (agent/session.ts) for the self-correction loop; deterministic caps fence it — emitting events back to Supabase the whole way so the UI animates live.

Daytona workspace

The agent's home — Opus 4.8 + the filesystem

Persistent sandbox: same filesystem holds clips.json, ffmpeg, file-memory, and the media files in MEDIA_DIR. This is where the long-horizon loop actually runs.

server/agent-server.ts — the worker / host

Long-running process (npm run agent). On boot reclaims orphaned queued runs, subscribes to runs INSERT, and atomically claims each one (win/lose). On a win it: consult() memory → runs the Agent SDK session below → uploads the shipped render to Storage renders → sets runs.shipped_render → distill() one rule on pass. Tools toggle via AGENT_TOOLS (stub = scripted money-shot · real = Claude + ffmpeg).

agent/session.ts — autonomous Agent SDK session (Claude drives; caps fence)

Claude owns iteration and calls the four MCP tools below in a loop; the session owns termination via maxTurns + maxBudgetUsd + wall-clock — never a free-running /goal with no cap. Caps: iters ≤ 4 · token budget · wallclock. tracker.ts rolls turns into a best_render + LoopResult. (The legacy deterministic loop/runLoop is test-only — this session replaced it as the loop owner.)

How to read the three pieces you asked about
Autonomous session = agent/session.ts. Claude owns the search→build→render→grade→fix cycle via MCP tools; the session owns caps + best-so-far. agent-server.ts is the worker that hosts it.
Daytona = where the worker + Opus 4.8 + ffmpeg + file-memory + media all live on one filesystem. The agent's persistent home; the worker runs as npm run agent.
Supabase = the decoupling bus, both ways. UI enqueues a runs row → worker claims via Realtime → worker writes events back. Neither side blocks the other — that's what makes the live scorecard reconnection-safe.
The split that matters: blue steps = agent() (judgment); ink steps = plain JS (render, validate, caps). Termination is trustworthy because the caps are code (maxTurns/maxBudgetUsd/wall-clock), even though the agent drives.
